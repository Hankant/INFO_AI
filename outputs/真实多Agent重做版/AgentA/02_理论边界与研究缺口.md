# 02 理论边界与研究缺口

## 先给结论

本项目如果被写成 `trust in AI`、`AI adoption`、`single-shot advice-taking` 或 `confidence calibration`，就已经降级了。  
它必须被写成：`human-AI collaboration architecture learning under repeated feedback`。

## 一、研究对象到底是什么

### 1. 正面定义

- 研究对象不是“AI 的答案会不会被采纳”，而是“AI 在决策系统中应占据什么角色”。
- 这里的 `architecture` 指的是流程中的角色配置，而不是模型参数或界面样式。
- 最低限度上，architecture 至少要区分：`human-only`、`AI advice`、`AI recommendation + human confirmation`、`AI default + human override`、`AI autonomous decision`。
- 真正要解释的是：人在重复接收绩效反馈之后，为什么会逐步收敛到其中某一类角色配置。

### 2. 最小理论链条

`重复绩效反馈 -> architecture attraction updating -> AI 与 human error 的非对称更新 -> 路径依赖 -> 长期收敛 -> 低 AI autonomy 锁定`

### 3. 核心命题

- 人不仅学习 `AI 准不准`。
- 人更在学习 `AI 配不配占据这个流程位置`。
- 因此，AI error 的制度含义大于单次建议错误本身。

## 二、它不是什么

### 1. 不是 trust 问题

- trust 问的是：我相不相信 AI 这次是对的。
- 本项目问的是：我是否还愿意把某种系统角色继续交给 AI。
- trust 可以是次级机制或控制变量，但不能是主因变量。

### 2. 不是 adoption 问题

- adoption 问的是：用不用 AI。
- 本项目问的是：即便用了 AI，它被放在什么位置，拥有多大 autonomy，保留多少 override 权。
- 只要你的理论可以被二元变量 `use AI / do not use AI` 充分描述，就说明你还没进入 architecture learning。

### 3. 不是单次 advice-taking 问题

- advice-taking 关心单一建议在单一回合是否被接受。
- 本项目关心的是多轮反馈中对“协作结构”的更新。
- 如果把整个实验压缩成一轮，命题仍然完整成立，那就不是本项目。

### 4. 不是 confidence calibration 问题

- calibration 关心的是人能否根据模型置信度线索更准确地判断某次输出值不值得信。
- 本项目关心的是人如何重配制度化控制权。
- 前者是 `signal-level learning`，后者是 `architecture-level learning`。

### 5. 不是一般性 complementarity 问题

- complementarity 文献问的是“人机结合什么时候比单方更好”。
- 本项目问的是“人在什么学习路径下会收敛到何种结合方式，且这种收敛是否偏离客观最优架构”。
- 我们不是在证明互补性存在，而是在解释为什么互补性常常实现不了。

## 三、真正的新意在哪里

### 1. 新意一：把 AI 错误从心理事件提升为制度冲击

- 旧写法：AI 出错会降低信任。
- 新写法：AI 出错会被解释为“AI 不应占据当前流程角色”，从而改变对整个 architecture 的吸引力评估。

### 2. 新意二：把结果变量从 reliance 改成 convergence

- 旧写法：人在某一轮更少依赖 AI。
- 新写法：人在多轮反馈后收敛到更低 AI autonomy 的稳定结构，且这种稳定结构未必客观最优。

### 3. 新意三：把动态重点放在 early error 的路径依赖

- 旧写法：AI error 的平均效应有多大。
- 新写法：当 AI error 出现在学习早期时，它会不会通过更强的负向 updating 改写后续可达路径，并造成 lock-in。

### 4. 新意四：把 EWA 从装饰性模型变成理论语言

- 旧写法：实验做完后再用一个动态模型拟合。
- 新写法：从一开始就把不同 architecture 看作会被反馈不断更新 attraction 的策略集合。

## 四、研究缺口必须这样表述

### 1. 第一层缺口：现有文献大多停留在单轮响应

- trust、adoption、advice-taking、delegation 文献，大都解释“这一轮怎么选”。
- 它们没有回答“长期会收敛到哪里”。

### 2. 第二层缺口：现有文献很少把 error source 与 error timing 放在一起

- 现有研究会讨论 AI error 比 human error 更刺眼。
- 但它们较少检验：这种非对称惩罚是否在早期更致命，并通过后续较少暴露、较慢恢复形成路径依赖。

### 3. 第三层缺口：现有文献很少同时比较主观收敛与客观最优

- 现有研究常问“人有没有正确依赖 AI”。
- 本项目必须再进一步：即便 AI 与 human 长期平均表现相同，甚至某种中等授权架构客观最优，参与者仍可能锁在低授权结构中。

### 4. 第四层缺口：现有文献缺少“角色学习”语言

- 现有文献多研究对能力的学习。
- 本项目研究对角色的学习。
- 这是从 `learning about AI capability` 走向 `learning AI's institutional role`。

## 五、三条硬判别标准

### 判别一

如果主因变量可以用一套 trust 量表完全替代，项目降级。

### 判别二

如果没有 repeated feedback，只剩一次采纳或一次 delegation，项目降级。

### 判别三

如果删掉 early error / late error 的时间结构，理论不受影响，项目降级。

## 六、最容易把项目写坏的地方

### 坏法一

“我们研究 AI 犯错后，人们是否更不信任 AI。”

问题在于：这只是 algorithm aversion 的近亲，不是 architecture learning。

### 坏法二

“我们研究高风险场景下，人是否更愿意保留 human-in-the-loop。”

问题在于：这只是既定流程偏好比较，还不是流程如何被学出来。

### 坏法三

“我们研究 AI 建议会不会让人过度依赖。”

问题在于：这仍是单轮 reliance，不是长期收敛。

### 坏法四

“我们用 EWA 拟合参与者选择，因此研究具有动态性。”

问题在于：如果 EWA 只是后验拟合，它没有承担理论识别任务。

## 七、能把项目保住的写法

### 好法一

“本研究将不同人机协作架构视为重复反馈中被持续更新 attraction 的策略集合，而非一次性流程偏好。”

### 好法二

“我们关心的不是 AI error 是否降低当次 reliance，而是它是否以更高负向权重改写对 AI autonomy 的长期授权路径。”

### 好法三

“核心结果不是 trust 下降，而是低 AI autonomy steady state 的路径依赖式锁定。”

### 好法四

“本文比较的不是 AI 与 human 谁更准，而是在长期平均表现相同的前提下，error source 与 timing 如何改变架构学习的收敛轨迹。”

## 八、建议直接采用的贡献句

- 第一，本文将 human-AI collaboration 从静态 reliance 问题推进为动态 architecture learning 问题。
- 第二，本文提出 AI error 的关键后果不是短期信任波动，而是通过非对称 architecture attraction updating 改变长期收敛路径。
- 第三，本文说明早期 AI error 可能导致低 AI autonomy 的路径依赖式锁定，即便这种稳态偏离客观最优协作架构。

## 九、必须主动承认的边界

- 本研究识别的是个体或小型监督者层面的 `micro-foundation of architecture learning`，不是直接证明真实组织层面的制度演化全貌。
- 本研究讨论的是可重复反馈环境中的角色学习，不是一次性不可重复高层战略决策。
- 本研究可以为组织治理提供机制基础，但不能直接声称已经观察到宏观制度锁定。

## 十、导师最终口径

一句话记住：  
`不是人在学会信不信 AI，而是在学会 AI 应该被放在系统的哪里；AI error 的危险不在于让人一时不信，而在于让人长期把 AI 锁在更低 autonomy 的位置。`
